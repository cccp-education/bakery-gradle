== Overview

Bakery is a Gradle plugin for static site generation using https://jbake.org[JBake]
with advanced features including https://www.thymeleaf.org[Thymeleaf] templating,
https://firebase.google.com[Firebase] authentication and comments,
https://www.google.com/forms/about/[Google Forms] embedding,
full-text search via https://pagefind.app[Pagefind],
AI-assisted article generation via https://ollama.com[Ollama] +
https://docs.langchain4j.dev[LangChain4j],
deployment to GitHub Pages via https://github.com/eclipse-jgit/jgit[JGit],
and the *Pattern LENTILLE* augmented context pipeline for intelligent cross-linking.

The plugin bundles JBake and all necessary dependencies (including AsciidoctorJ
diagrams with PlantUML), eliminating the need for manual JBake installation.

== Features

- *JBake*: Static site generation from AsciiDoc and Markdown content
- *Thymeleaf*: Modern template engine with extended expression support
- *Firebase*: Authentication (Google/email) + Firestore-based comments
- *Google Forms*: Contact form embedding with Firestore schema
- *Pagefind*: Full-text static search with zero infrastructure
- *AI Article Generation*: Blog post generation via Ollama LLM
- *Theme System*: Configurable theme catalog with variant resolution
- *IA Scaffolding*: Site structure generation from natural language intention
- *Pattern LENTILLE*: Augmented context pipeline (segregation → enrichment → budget → injection)
- *AsciidoctorJ Diagrams*: PlantUML rendering in AsciiDoc content
- *JGit Deployment*: Git-based deployment to GitHub Pages or any Git remote
- *5-Layer Config Cascade*: CLI → gradle.properties → DSL → site.yml → defaults
- *Arrow-kt*: Functional programming with `Either`, `Option`, `Raise DSL`
- *Analytics & Newsletter*: Configurable Firebase Analytics + newsletter integration
- *i18n*: Internationalization with 10 languages (English, Chinese, Hindi, Spanish, French, Arabic, Bengali, Portuguese, Russian, Urdu) via Thymeleaf message externalization

=== Configuration Cascade

The plugin resolves configuration values through a 5-layer cascade (highest to lowest priority):

1. *CLI*: `-Pbakery.config.path=/path/to/site.yml`
2. *gradle.properties*: `bakery.config.path=...`
3. *DSL*: `bakery { configPath = ... }` in build script
4. *site.yml*: YAML configuration file
5. *Defaults*: Built-in sensible defaults

=== Security

*IMPORTANT*: Never hardcode credentials. Use environment variables or GitHub Secrets:

[source,yaml]
----
password: "${GITHUB_TOKEN:-your-preview-token}"
apiKey: "${FIREBASE_API_KEY}"
----

== Quick Start

=== 1. Initialize a New Site

[source,bash]
----
./gradlew initSite
----

This creates a `site.yml` configuration file and copies default `site/` (JBake content, templates, assets) and `maquette/` directories.

=== 2. Build the Site

[source,bash]
----
./gradlew bake
----

Generates the static site in `build/bake/`.

=== 3. Serve Locally

[source,bash]
----
./gradlew serve
----

Serves the site at `http://localhost:8820` with live reload on changes.

=== 4. Deploy to GitHub Pages

[source,bash]
----
./gradlew deploySite
----

=== 5. Generate an Article with AI (Optional)

Enable IA in `site.yml`:

[source,yaml]
----
ia:
  enabled: true
  baseUrl: "http://localhost:11464"
  modelName: "deepseek-v4-pro"
----

Then:

[source,bash]
----
./gradlew generateArticle
    -Pbakery.article.topic="Introduction to Gradle Plugins"
    -Pbakery.article.tone="professional"
    -Pbakery.article.audience="Java developers"
    -Pbakery.article.keywords="gradle, plugins, kotlin"
    -Pbakery.article.language="en"
----

=== 6. Scaffold a Site from Intention (Optional)

[source,bash]
----
./gradlew generateSiteFromIntention
    -Pbakery.scaffold.description="A developer blog with projects and about pages"
    -Pbakery.scaffold.siteType="BLOG"
    -Pbakery.scaffold.lang="en"
    -Pbakery.scaffold.projectName="my-dev-blog"
----

== Pattern LENTILLE — Augmented Context Pipeline

The *Pattern LENTILLE* enriches site content with context from the workspace
knowledge graph and RAG (Retrieval-Augmented Generation). The pipeline runs in
six steps:

1. *Load*: Composite context from `graphify-plugin` knowledge graph
2. *Extract*: Subgraph relevant to current content
3. *RAG*: Retrieve related content via vector search
4. *Score*: Hybrid scoring (RAG + KG + tags + cross-references + BFS proximity)
5. *Filter*: Apply rules + budget constraints
6. *Inject*: Serialize augmented context JSON for Thymeleaf templates

=== DSL Configuration

.build.gradle.kts
[source,kotlin]
----
bakery {
    augmentedContext {
        enabled = true
        compositeContextPath = file("composite-context.json").absolutePath
        budget {
            maxArticlesPerPage = 5
            priorityOrder = listOf("TAG_MATCH", "RAG_SEMANTIC", "KG_PROXIMITY")
        }
        rules {
            excludeTags = listOf("archived", "draft")
            requireCrossReference = true
        }
    }
}
----

=== Augmented Articles Template

In Thymeleaf templates, access augmented context via:

[source,html]
----
<div th:if="${augmentedArticles != null and !augmentedArticles.isEmpty()}">
    <h2>Related Articles</h2>
    <div th:each="article : ${augmentedArticles}">
        <a th:href="${article.uri}" th:text="${article.title}">Article Title</a>
        <span class="badge" th:each="source : ${article.sources}"
              th:text="${source}">RAG</span>
    </div>
</div>
----

== Internationalization (i18n)

The plugin supports 10 languages via Thymeleaf message externalization.
Each language has a `messages_{code}.properties` file with ~65 translated keys.

=== Supported Languages

[cols="1,3,2"]
|===
| Code | Language | Direction
| `en` | English | LTR
| `zh` | Chinese Mandarin (中文) | LTR
| `hi` | Hindi (हिन्दी) | LTR
| `es` | Spanish (Español) | LTR
| `fr` | French (Français) — default | LTR
| `ar` | Arabic Standard (العربية) | RTL
| `bn` | Bengali (বাংলা) | LTR
| `pt` | Portuguese (Português) | LTR
| `ru` | Russian (Русский) | LTR
| `ur` | Urdu (اردو) | RTL
|===

=== Configuration

The active language is resolved through the 5-layer cascade (default: `fr`):

[source,bash]
----
# CLI (highest priority)
./gradlew bake -Pbakery.language=en

# gradle.properties
bakery.language=ar

# DSL
bakery {
    language = "zh"
}

# site.yml
language: hi
----

The resolved language is injected as `site.language` in `jbake.properties`,
exposed to templates as `${config.site_language}`.

=== Template Usage

Templates use `#{key}` for internationalized text:

[source,html]
----
<a class="nav-link" th:href="..." th:text="#{nav.home}">Home</a>
<h1 th:text="#{nav.blog}">Blog</h1>
----

The `<html>` tag automatically sets `lang` and `dir` attributes:

[source,html]
----
<html th:lang="${config.site_language}"
      th:dir="${#sets.contains({'ar','ur'}, config.site_language) ? 'rtl' : 'ltr'}">
----

=== Adding a New Language

. Create `messages_{code}.properties` in `src/main/resources/site/templates/`
. Translate all ~65 keys
. Add the code to `SUPPORTED_LANGS` in `ScaffoldIntention.kt`

== CI/CD with GitHub Actions

The project provides three GitHub Actions workflows.

=== Test Workflow

`.github/workflows/test.yml` — Runs on every push and PR:

- Compiles with JDK 24, Kotlin {kotlin-version}, Gradle {gradle-version}
- Runs unit tests (JUnit5), functional tests (TestKit), Cucumber scenarios
- Installs Playwright Chromium and runs E2E browser tests
- Verifies Kover code coverage >= 70%

=== Website Deploy

`.github/workflows/website.yml` — Deploys site on push to `main`:

.Required secret: `BAKERY_GRADLE_PLUGIN` containing the full `site.yml` content.
.The workflow injects this secret → `site.yml` → runs `deploySite`.

=== README Generation

`.github/workflows/readme_plantuml.yml` — Generates README with PlantUML diagrams
on push to `main` when `README_plantuml*.adoc` files change, using the
`readme-gradle` plugin.

=== Secret Configuration

[arabic]
. Navigate to your GitHub repository *Settings* > *Secrets and variables* > *Actions*
. Click *New repository secret*
. Name: `BAKERY_GRADLE_PLUGIN`
. Secret: entire `site.yml` content

== Troubleshooting

=== Plugin Not Found

- Ensure `mavenLocal()` and `gradlePluginPortal()` are in `pluginManagement.repositories`
- Verify the plugin ID is `{plugin-id}`
- Check version `{plugin-version}` in `libs.versions.toml`

=== Site Not Deploying

- Verify GitHub credentials in `site.yml`
- Ensure your GitHub token has `repo` and `workflow` permissions
- Check that the target repository and branch exist
- Run `./gradlew verifyConfigurationMapping` to validate configuration

=== Local Server Issues

- Ensure port 8820 is not in use
- Verify the `site/` directory exists with valid JBake content
- Check JBake properties in `site/jbake.properties`

=== AI Features Not Working

- Ensure Ollama is running on the configured endpoint (default `http://localhost:11464`)
- Verify `ia.enabled = true` in `site.yml`
- Check the model name matches an available Ollama model

== Architecture

The plugin follows clean architecture principles with functional programming
via https://arrow-kt.io[Arrow-kt]:

- *Task Registrars*: `SiteTaskRegistrar`, `DeployTaskRegistrar`, `ContentTaskRegistrar`, `LensTaskRegistrar`
- *ConfigResolver*: 5-layer configuration cascade with proper precedence
- *Pattern LENTILLE*: 6-step augmented context pipeline
- *Arrow-kt `Either`*: Domain error handling without exceptions
- *Dependencies*: Aligned on `{group-id}:workspace-bom:{bom-version}` (MEMPHIS BOM)
- *AI*: LangChain4j Ollama integration with configurable timeout and pool

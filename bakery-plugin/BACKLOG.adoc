== EPIC BKY-TREE — Arbre de Site : Composite Site→Section→Article→AST

_Cadré Session 165 (2026-06-25). Réouvre le backlog bakery après confirmation
production-ready (S164). Concept unifiant manquant qui relie PivotModel (AST
AsciiDoc, S157-163), SiteConfiguration (config plate), ScaffoldOutput (liste
plate de templates), et la Loi de l'Économie d'Encre (delta global → delta par
sous-arbre)._

[CAUTION]
====
**Motivation** : `SiteConfiguration` est plate (infra only : push, firebase,
theme). `ScaffoldOutput.templates: List<String>` est plate. La migration i18n
walk `templates/` à plat. Aucun concept ne modélise l'arborescence éditoriale
du site (section → article → contenu). Conséquence : la Loi de l'Économie
d'Encre ne peut pas calculer un delta ciblé par sous-arbre — elle opère
globalement, re-traduisant plus que nécessaire.

**Vision** : introduire un agregat `SiteTree` (Composite pattern) à 4 niveaux —
`Site` (root) → `Section` → `Article` (feuille) → `PivotModel` (AST déjà
construit S157-163). Chaque niveau manipulable indépendamment : theme hérité
ou surchargé, i18n par sous-arbre, accessibilité par section, Économie d'Encre
par article (delta sur l'AST, pas sur tout le site).

**Pré-requis respectés** :
. *Ne pas réinventer PivotModel* — il est fait, testé, golden-master. `Content`
  l'enveloppe comme feuille, point.
. *Backward compat* — `ScaffoldOutput.templates: List<String>` reste accepté
  via adapter flatten. Le nouveau `tree:` est additif dans `site.yml`.
. *Scope* — ~36 pts sur 7 US, à slicer sur 2-3 sessions minimum.
. *Méthodo* — baby steps DDD (sealed interface pur, testable sans Gradle) →
  TDD (UT rouge avant code) → BDD (feature Cucumber pour bout en bout). Aucune
  I/O dans les classes de domaine.
====

=== Arbre à 4 niveaux

----
Site (root)                    ← jbake.properties, theme global, i18n config
└── Section                    ← template de section, theme override (optionnel)
    └── Article                ← frontmatter + AsciiDoc source
        └── PivotModel (AST)  ← déjà construit (S157-163)
            ├── Heading, Paragraph, List, Source, Table, Admonition...
            └── inline: Bold, Code, Link, Text
----

== EPIC BKY-I18N-REAL — Migration i18n réelle des 3 sites sur copies (golden masters post-migration)

_Cadré Session 170 (2026-06-29). La tuyauterie `I18nMigrationService` +
`MigrateToI18nTask` existe et fonctionne (tests d'intégration GREEN sur
`@TempDir`). Les 3 fixtures pré-migration sont alignées (S150), les

post-migration persistant n'existe — la migration n'a jamais été figée

les fixtures (uniquement `translations_en.properties` séparé)._

[CAUTION]
====
**Motivation** : les 3 tests d'intégration (`MagicStickI18nMigrationIntegrationTest`,
`CccpEducationI18nMigrationIntegrationTest`, `CherolivComI18nMigrationIntegrationTest`)
valident que la migration *fonctionne* sur copies temporaires, mais le
résultat n'est jamais persisté. Conséquence : aucune régression sur le
rendu post-migration ne peut être détectée, et l'idempotence (Loi de
l'Économie d'Encre) n'est pas testée sur données réelles.

**Vision** : introduire un concept `PostMigrationSnapshot` (DDD pur,
testable sans Gradle) qui capture l'état d'un site après migration
(templates migrés + `messages_fr.properties` + `messages_en.properties`
rempli). Des golden masters persistants (un par site) figent le résultat
attendu. Un test TDD rouge assert byte-identical. Une feature BDD Cucumber
valide bout en bout. L'idempotence est testée : re-execute migrate sur
copie déjà migrée = zéro changement.

**Pré-requis respectés** :
. *Ne pas toucher à la production* — tout sur copies (`@TempDir` ou
  `src/test/resources/i18n-fixtures/{site}/post-migration/`).
. *Backward compat* — les fixtures pré-migration restent intactes (nouveaux
  snapshots sont additifs dans `post-migration/`).
. *Méthodo* — baby steps DDD (concept pur, testable sans Gradle) → TDD

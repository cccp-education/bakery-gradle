= PARTIE II — Brainstorm : Bakery Autonome + Intégré dans l'Écosystème Workspace

[abstract]
--
Session 022, seconde phase. L'audit factuel a révélé un plugin mature sur le plan
mécanique (build, test, déploiement) mais totalement absent sur le plan IA.
L'objectif de cette partie est de projeter bakery dans le concert du workspace :
quels workflows IA-assistés doit-il embarquer POUR LUI-MÊME (autonomie),
et quels contrats doit-il exposer POUR LES AUTRES (intégration DAG N2 → N3) ?

La thèse : *bakery n'est pas juste un générateur de site statique. C'est le point
de rendez-vous où le contenu (articles, slides, SPDs, formulaires, thèmes) rencontre
le déploiement. Sa mission dans le MVP0 formation est d'être le dernier maillon N2
avant l'orchestration N3 — celui qui matérialise le livrable visible.*
--

'''
== Règles Architecturale du Brainstorm

. *DAG N2* : bakery NE PEUT PAS dépendre de N1 (codebase-gradle). Il peut dépendre
  de N0 (graphify-gradle) et de pgvector (shared service). Il est consommé par N3
  (runner-gradle).
. *DOGFOOD FIRST* : toute fonctionnalité IA est utilisée par bakery en interne avant
  d'être exposée à l'utilisateur final. Le pattern clasp (privé) est l'exemple canonique.
. *Contrat N3* : bakery doit produire un `metadata.json` structuré pour que runner-gradle
  puisse l'agréger dans le vecteur composite. Sans ce contrat, bakery est invisible
  pour l'orchestrateur.
. *Ne pas réinventer* : slider-gradle et plantuml-gradle ont déjà des patterns
  LangChain4j + Ollama + prompt engineering qui fonctionnent en production. Bakery
  doit les adapter, pas les refaire.

'''
== Analyse des Axes Proposés

=== Axe 1 — Scaffolding Assisté par IA

*Proposition* : le `generateSite` actuel est mécanique (copie de templates
JBake fixes). L'IA pourrait choisir la structure du site, les templates, les
métadonnées JBake en fonction d'une intention exprimée en langage naturel.

*Analyse* :
- EPIC 10 (Scaffolding TDD) est à 80% — la mécanique de fondation est presque
  prête. L'ajout d'un prompt LLM au-dessus du scaffolding est une couche
  mince une fois la base solide.
- Le pattern existe déjà : plantuml-gradle fait "décris ton diagramme en langage
  naturel → je génère le PlantUML". Bakery peut faire "décris ton site → je
  génère la structure JBake + site.yml".
- Le LLM n'a pas besoin de connaître JBake — il a besoin de connaître les
  *intentions éditoriales* (blog, portfolio, formation, documentation) et bakery
  fait le mapping intention → structure.

*Verdict* : ✅ *Validé — P0* une fois EPIC 10 terminé. Charge : ~5 pts.
Découpage :
. `BKY-IA-1.1` — Ajouter `BakeryExtension.ia { prompt { } }` dans le DSL
. `BKY-IA-1.2` — `ScaffoldIntention` data class (type, langue, audience, sections)
. `BKY-IA-1.3` — Tâche `generateSiteFromIntention` qui prompte le LLM →
  structure → scaffolding
. `BKY-IA-1.4` — Tests unitaires + Cucumber (intention → structure valide)

=== Axe 2 — Personnalisation de Thème Assistée par IA

*Proposition* : génération/modification de templates Thymeleaf + CSS en fonction
d'un domaine visuel (couleurs, typographie, mise en page).

*Analyse* :
- Bakery utilise Thymeleaf comme moteur de template. Les templates sont dans
  `src/main/resources/site/templates/`. Le LLM peut générer des variantes.
- C'est un axe puissant MAIS risqué : la génération de CSS/HTML par LLM est
  notoirement instable (hallucinations de classes, CSS invalide, design incohérent).
- Approche recommandée : *paramétrique plutôt que générative*. Le LLM choisit
  parmi un catalogue de thèmes prédéfinis + ajuste des variables (couleurs,
  polices, espacement) plutôt que de générer du CSS from scratch.
- Pattern : slider-gradle fait déjà ça avec les thèmes reveal.js — le LLM
  sélectionne un thème existant, ne génère pas de CSS.

*Verdict* : ⚠️ *Validé avec contrainte — P1*. Approche paramétrique (catalogue
de thèmes + variables) plutôt que générative (CSS from scratch).
Charge : ~8 pts. Découpage :
. `BKY-IA-2.1` — Catalogue de thèmes Thymeleaf (5-10 variantes : minimal, magazine,
  documentation, portfolio, formation)
. `BKY-IA-2.2` — Variables de thème (couleurs, polices, logo, favicon)
. `BKY-IA-2.3` — Tâche `generateTheme` : intention → sélection thème + variables
. `BKY-IA-2.4` — Tests : vérifier que les templates générés compilent avec Thymeleaf

=== Axe 3 — Connexion à Firebase Assistée par IA

*Proposition* : le LLM génère la configuration Firebase (Firestore rules,
collections, cloud functions) et l'injecte dans le site.

*Analyse* :
- EPIC 4 (Formulaire de contact Firebase) est déjà à 95%. Le flux manuel est
  documenté : `site.yml` → `SiteConfiguration.kt` → injection dans `jbake.properties`
  → template Thymeleaf `footer.thyme` → SDK Firebase dans le HTML.
- L'IA n'ajoute pas grand-chose au flux actuel qui est déjà bien mécanisé.
  La vraie valeur IA serait de *générer le backend Firebase* (Firestore rules,
  schéma de collection, cloud functions pour traitement des formulaires) —
  mais ça sort du scope de bakery. C'est du backend-as-code, pas du site statique.
- Ce que bakery PEUT faire : détecter qu'un formulaire est configuré dans `site.yml`
  → proposer de générer la config Firebase via LLM → injecter. Mais le gain
  est marginal par rapport au flux actuel.

*Verdict* : ⚠️ *Partiellement validé — P2*. Le flux Firebase actuel est déjà
opérationnel. L'IA ajouterait de la valeur sur la *validation* (vérifier que
la config Firebase est cohérente) plutôt que sur la *génération*.
Charge : ~3 pts. Découpage :
. `BKY-IA-3.1` — Tâche `validateFirebaseConfig` : le LLM vérifie la cohérence
  du bloc `firebase:` dans `site.yml` (projectId valide, règles cohérentes)
. `BKY-IA-3.2` — Suggester : "tu as un formulaire de contact, veux-tu aussi un
  formulaire de newsletter ?"

=== Axe 4 — Génération d'Article Assistée par IA

*Proposition* : tâche `generateArticle` qui prend un topic + un corpus d'articles
existants → produit un article AsciiDoc avec métadonnées JBake.

*Analyse* :
- C'est *la killer feature pour cheroliv.com*. Le blog est le principal
  consommateur de bakery aujourd'hui. Pouvoir dire "génère-moi un article
  sur pgvector avec des exemples Kotlin" et obtenir un `.adoc` prêt à baker
  est une accélération massive du pipeline éditorial.
- Bakery a déjà `ConfigPrompts.kt` (prompts interactifs). Le pattern peut être
  étendu : prompts → intention éditoriale → LLM → article AsciiDoc.
- Le corpus existe : les articles déjà publiés sur cheroliv.com sont la
  mémoire éditoriale. Le LLM peut s'appuyer sur le style, le ton, la structure
  existants.
- Le contrat DAG est respecté : bakery appelle le LLM directement (LangChain4j
  + Ollama) ou via pgvector (shared service). Pas de dépendance à codebase-gradle.
- *Point critique* : la qualité éditoriale. Un article généré par IA sans
  relecture humaine est un risque. Le workflow doit être : ébauche IA →
  relecture humaine → publication. Bakery fournit l'ébauche, pas le produit fini.

*Verdict* : ✅ *Validé — P0*. C'est l'axe qui apporte le plus de valeur
immédiate à l'utilisateur principal (cheroliv.com). Charge : ~8 pts.
Découpage :
. `BKY-IA-4.1` — `ArticleIntention` data class (topic, audience, ton, longueur,
  mots-clés, articles similaires)
. `BKY-IA-4.2` — Tâche `generateArticle` : intention → prompt LLM structuré →
  article AsciiDoc avec header JBake (`:jbake-title:`, `:jbake-tags:`, etc.)
. `BKY-IA-4.3` — Intégration LangChain4j + Ollama (DeepSeek-v4-pro) — même
  pattern que slider-gradle et plantuml-gradle
. `BKY-IA-4.4` — Prompt engineering : template de prompt éditorial qui force
  le style, la structure, et les métadonnées JBake
. `BKY-IA-4.5` — Tests : vérifier que l'article généré est un AsciiDoc valide,
  que le header JBake est parsable, que le contenu n'est pas vide

=== Axe 5 — Formulaire Google Forms : Bakery → Edster → CI/CD → Drive Utilisateur

*Proposition (v3 — corrigée après feedback utilisateur)* : une tâche bakery
prépare la recette du formulaire, l'API REST Edster provisionne le workflow CI/CD,
et c'est la CI/CD de l'utilisateur qui exécute clasp pour livrer le formulaire
dans son Drive. Bakery récupère l'URL finale et l'injecte dans le site.

*Analyse* :
- *Architecture en 3 couches* :
  . *Bakery (N2)* — génère la **recette** (intention IA → code Apps Script `Code.gs`
    + `appsscript.json` + `FormIntention`). Ne sait RIEN d'Edster. Ne connaît que
    son contrat N2 → N3 : produire un artefact (`build/bakery/forms/<formId>/recipe.json`).
  . *Edster (N3, SaaS)* — reçoit la recette via l'API REST (`POST /provision/form`),
    provisionne le job CI/CD (GitHub Actions) dans le repo de l'utilisateur.
    C'est le SaaS — il tourne sur le VPS OVH (edster.cloud, 30€/mois, marge ~95%).
  . *CI/CD utilisateur* — le job GitHub Actions exécute clasp (OAuth scope Drive
    utilisateur), déploie le formulaire dans le Drive, renvoie l'URL du formulaire
    publié à Edster (callback ou webhook).
  . *Bakery (N2) — récupération* — runner-gradle (N3) récupère l'URL depuis Edster,
    appelle bakery pour injecter l'URL dans la page du site (`deployFormUrl`).

- *Contrat DAG strictement respecté* :
  - Bakery (N2) NE PEUT PAS appeler Edster (N3) directement → violation DAG.
  - Le flux est : Bakery → runner-gradle (N3) → Edster (N3 SaaS) → CI/CD user.
  - Bakery expose une tâche `GENERER` (`generateFormRecipe`) qui produit la recette.
  - Runner-gradle orchestre : récupère la recette → appelle Edster → attend le
    callback → appelle bakery pour l'injection.

- *Flux complet* :
----
1. Utilisateur : site.yml → forms: { type: "evaluation", questions: 5, googleAccount: "formateur@gmail.com" }
2. Bakery : generateFormRecipe → LLM → Code.gs + appsscript.json + FormIntention
   → build/bakery/forms/<formId>/recipe.json
3. Runner-gradle (N3) : collecte recipe.json → POST /provision/form → Edster
4. Edster : provisionne le job CI/CD (GitHub Actions) dans le repo de l'utilisateur
   → injecte les secrets (CLASP_TOKEN, GOOGLE_ACCOUNT)
5. CI/CD utilisateur : clasp login → clasp push → clasp deploy → callback Edster avec URL
6. Edster → runner-gradle : URL du formulaire publié
7. Runner-gradle → bakery : deployFormUrl(url, page="evaluation.thyme")
8. Bakery : injecte l'URL dans la page Thymeleaf → bake → deploySite
----

- *L'IA intervient à un seul niveau — la recette* :
  . "Je veux un formulaire d'évaluation avec 5 questions QCM" → LLM → JSON structuré
    → code Apps Script. C'est bakery qui génère, pas Edster. Edster ne fait
    qu'orchestrer le déploiement.

- *Séparation des responsabilités* :
  [cols="1,3", options="header"]
  |=== 
  | Composant | Responsabilité
  | Bakery (N2) | Générer la recette (code + intention). Injecter l'URL finale
    dans le site. Ne connaît PAS Edster.
  | Runner-gradle (N3) | Orchestrer le flux : collecter recette → appeler Edster
    → récupérer URL → appeler bakery.
  | Edster (N3 SaaS) | Provisionner le job CI/CD. Gérer les credentials utilisateur.
    Exposer l'API REST. Facturer.
  | CI/CD utilisateur | Exécuter clasp. Posséder le token OAuth. Livrer le
    formulaire dans le Drive.
  |===

- *Pourquoi cette architecture* :
  - *DOGFOOD FIRST, SAAS LATER* — en mode dogfood (MVP0), le job CI/CD est celui
    du développeur (cheroliv). En mode SaaS (MVP1), Edster provisionne le job
    chez le client.
  - *Sécurité* : le token OAuth Google (CLASP_TOKEN) n'est jamais chez bakery.
    Il est injecté comme secret GitHub Actions dans la CI/CD de l'utilisateur.
  - *RGPD* : le formulaire est dans le Drive de l'utilisateur. Les réponses
    sont dans son Google Sheets. Bakery ne voit aucune donnée stagiaire.
  - *Monétisation* : Edster facture le provisionnement. C'est la brique
    transactionnelle (waiter-gradle). Bakery reste Apache 2.0 gratuit.

*Verdict* : ✅ *Validé — P0 pour le MVP0 formation*. Bakery est le générateur
de recette, Edster est le provisionneur, la CI/CD utilisateur est l'exécuteur.
Charge : ~8 pts (réduit car le déploiement est délégué à Edster).
Découpage :
. `BKY-IA-5.1` — `FormIntention` data class (type : contact, évaluation, inscription,
  feedback ; questions : QCM, texte, échelle ; googleAccount cible)
. `BKY-IA-5.2` — `FormGenerator.kt` (GENERER) : intention → LLM → code Apps Script
  (`Code.gs` + `appsscript.json` + `recipe.json`)
. `BKY-IA-5.3` — Tâche `generateFormRecipe` : LLM prompting → recipe.json dans
  `build/bakery/forms/<formId>/`
. `BKY-IA-5.4` — Tâche `deployFormUrl` : reçoit l'URL du formulaire (depuis
  runner-gradle) → injecte dans la page Thymeleaf
. `BKY-IA-5.5` — Extension DSL `bakery { forms { googleAccount = "..." } }`
. `BKY-IA-5.6` — Tests : mock LLM → recipe.json valide, injection URL dans template
  Cucumber : génération recette → code Apps Script parsable, injection URL ok

*Note sur Edster* : l'API REST Edster (`POST /provision/form`) n'est PAS dans
le scope bakery. C'est le contrat runner-gradle ↔ Edster. Bakery ne connaît
que `recipe.json` (sortie) et `deployFormUrl` (réception). Le reste est
opaque pour bakery — c'est le principe du DAG.

*Stratégie MVP0 — Mock Edster* : en attendant qu'Edster soit opérationnel
(API REST + CI/CD provisioning + waiter-gradle facturation), le flux est mocké
au niveau runner-gradle :
- Runner-gradle lit `build/bakery/forms/<formId>/recipe.json`
- Au lieu d'appeler Edster, il exécute directement clasp en local (via `npx @google/clasp`)
  avec le token OAuth du développeur (cheroliv) → le formulaire est dans le Drive
  cheroliv. C'est le mode *dogfood* pur.
- La tâche `generateFormRecipe` (bakery) est réelle dès le jour 1 — c'est la
  génération IA qui est testée et validée, pas le déploiement Edster.
- Le switch vers Edster se fera en une seule ligne dans runner-gradle :
  remplacer `ClaspLocalExecutor` par `EdsterApiClient`. Le contrat recipe.json
  ne change pas.

Ce mock est aligné sur la règle d'or : *DOGFOOD FIRST, SAAS LATER*. Tant qu'il
n'y a pas de client externe, pas besoin d'API REST Edster — clasp local suffit.

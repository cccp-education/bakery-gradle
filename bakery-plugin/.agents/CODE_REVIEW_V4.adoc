== PROBLÈMES RÉSIDUELS — 9 Code Smells Inédits (V4)

=== 🔴 CS-V4-1 : BakeryPlugin.kt:79 — `Throwable` fantôme alloué inutilement (P1)

[source,kotlin,linenums]
----
project.logger.warn(
    "[bakery] configPath is not set — only scaffold tasks will be available. " +
        "Define it via DSL (`bakery { configPath = \"site.yml\" }`), " +
        "`gradle.properties` (`bakery.config.path=site.yml`), or `-Pbakery.config.path=...`.",
    Throwable("configPath absent — stack trace for debugging")  // ← ALLOUÉ MAIS JAMAIS LOGGÉ
)
----

*Problème* : `Logger.warn(String, vararg Any?)` accepte le `Throwable`
comme vararg `Object`, pas comme second paramètre `Throwable`.
Résultat : le stack trace n'est jamais loggé, mais l'allocation mémoire
et le fill de stack trace sont exécutés à chaque appel.

*Impact* : Performance (allocation inutile à chaque `apply()` sans configPath).
*Fix* : Supprimer le `Throwable(...)` — c'est un artefact de debug.
Ou utiliser `logger.warn("message", exception)` si vraiment nécessaire.

*Time* : 2 min.

=== 🟠 CS-V4-2 : VerifyConfigurationMappingTask.kt:44 — Path CWD, pas projectDir (P1)

[source,kotlin,linenums]
----
val path = configPath.get()
val configFile = File(path)  // ← Résolu relativement au CWD, pas au projectDir
----

*Problème* : Si l'utilisateur exécute `./gradlew verifyConfigurationMapping`
depuis un sous-répertoire, `File("site.yml")` cherchera dans le CWD, pas
dans `$projectDir/site.yml`.

*Impact* : `GradleException("Configuration file not found: site.yml")` alors
que le fichier existe à la racine du projet. UX dégradée.

*Fix* :
[source,kotlin]
----
val configFile = project.projectDir.resolve(path)
----

*Time* : 5 min. Déjà identifié dans `last_code_review.adoc` (CS-27) — non résolu.

=== 🟠 CS-V4-3 : SiteManager.configureConfigPath — 2 niveaux d'imbrication + `run` inutile (P2)

[source,kotlin,linenums]
----
} else {
    val gradlePropertiesFile = layout.projectDirectory.asFile.resolve("gradle.properties")
    if (gradlePropertiesFile.exists())
        properties.run {  // ← `run` est un scope function inutile — properties est un Map<String, *>
            val configPath = get(BakeryConstants.BAKERY_CONFIG_PATH_KEY)?.toString()
            if (keys.contains(BakeryConstants.BAKERY_CONFIG_PATH_KEY) &&
                !configPath.isNullOrBlank() &&
                configPath.isYmlUri
            ) bakeryExtension.configPath.set(configPath)
        } else logger.info(...)  // ← else rattaché au `if` externe, difficile à lire
}
----

*Problème* : Le `properties.run { }` n'apporte aucune valeur — c'est un
`Map<String, *>` en lecture seule, pas besoin de scope function. La triple
condition sur une ligne est illisible.

*Fix* :
[source,kotlin]
----
} else {
    val gradlePropertiesFile = layout.projectDirectory.asFile.resolve("gradle.properties")
    if (!gradlePropertiesFile.exists()) {
        logger.info("gradle.properties not found — DSL or CLI config required")
        return
    }
    val configPath = properties[BakeryConstants.BAKERY_CONFIG_PATH_KEY]?.toString()
    if (configPath.isNullOrBlank() || !configPath.isYmlUri) {
        logger.info("No valid configPath in gradle.properties")
        return
    }
    bakeryExtension.configPath.set(configPath)
}
----

*Time* : 10 min.

=== 🟡 CS-V4-4 : `DeployProfileTask` — coquille vide (P2)

[source,kotlin,linenums]
----
internal fun Project.registerDeployProfileTask(site: SiteConfiguration) {
    tasks.register("deployProfile", DeployProfileTask::class.java) { task ->
        task.apply {
            group = BakeryConstants.DEPLOY_GROUP
            description = "Push profile files (e.g. README.md) to GitHub repository"
        }
    }
}
----

*Problème* : La tâche est enregistrée mais **aucune propriété n'est configurée**.
Ni `site`, ni `configPath`, ni `profileFiles`. La tâche existera dans `./gradlew tasks`
mais s'exécutera sans effet (ou avec des valeurs par défaut vides).

*Vérification* : Lire `DeployProfileTask.kt` — quelles sont les `@Input` properties ?
Si elles sont `abstract val`, le build plantera. Si elles ont des defaults, la
tâche tournera dans le vide.

*Time* : 5 min + lire DeployProfileTask.

=== 🟡 CS-V4-5 : `configureBakeTask` — `getByName()` fragile (P2)

[source,kotlin,linenums]
----
internal fun Project.configureBakeTask(site: SiteConfiguration) {
    tasks.withType(JBakeTask::class.java)
        .getByName(BakeryConstants.BAKE_TASK).apply {  // ← getByName jette si absent
            group = BakeryConstants.GENERATE_GROUP
            ...
        }
}
----

*Problème* : `getByName()` lève `UnknownTaskException` si la tâche "bake"
n'existe pas. Comme cette méthode est appelée dans `registerFullPipelineTasks`
après `configureJBakePlugin` (qui applique le plugin JBake), c'est _normalement_
toujours présent. Mais si le plugin JBake change sa convention de nommage,
c'est un crash silencieux au runtime.

*Fix* : `findByName(BakeryConstants.BAKE_TASK)?.apply { ... } ?: logger.warn(...)`

*Time* : 3 min.

=== 🟢 CS-V4-6 : `BakeryExtension` — 12 méthodes `Action<X>` identiques (P3)

[source,kotlin,linenums]
----
fun googleForms(action: Action<GoogleFormsDsl>) { action.execute(googleForms) }
fun firebaseAuth(action: Action<FirebaseAuthDsl>) { action.execute(firebaseAuth) }
// ... 10 autres identiques
----

*Problème* : 12 méthodes qui suivent exactement le même pattern.
2 lignes × 12 = 24 lignes de boilerplate.

*Contre-argument* : C'est le pattern idiomatique Gradle, tous les plugins
officiels font ça. Ce n'est pas une dette technique, c'est une convention.

*Verdict* : **ACCEPTÉ** — idiomatique Gradle. Pas d'action.

=== 🟢 CS-V4-7 : `ConfigPromptM.fromCliOrPrompt` — nommage confus (P3)

La méthode s'appelle `fromCliOrPrompt` mais la logique interne vérifie
aussi les variables d'environnement (`System.getenv`). Le nom ne reflète
pas la 3ème source (env vars). `fromCliOrEnvOrPrompt` serait plus honnête
mais verbeux. `resolveOrPrompt` capture mieux l'intention.

*Time* : 3 min (rename-only).

=== 🟢 CS-V4-8 : `ContentTaskRegistrar` — `try-catch` muet sur `dsl.toIntention()` (P3)

[source,kotlin,linenums]

=== 🟢 CS-V4-9 : `escapeJsonForJavaProperties` — pas idempotente (P3)

[source,kotlin]
----
internal fun escapeJsonForJavaProperties(rawJson: String): String =
    rawJson
        .replace("\\", "\\\\")
        .replace("\n", "\\n")
        .replace("\r", "\\r")
        .replace("\t", "\\t")
----

*Problème* : Si par malheur le pipeline appelle `escapeJsonForJavaProperties()`
deux fois sur la même chaîne, le JSON est corrompu :
- Appel 1 : `{"key":"val"}` → `{"key":"val"}` (normal)
- Appel 2 : `\\` devient `\\\\`, `\n` devient `\\n`, etc.

En pratique, le pipeline actuel ne fait jamais de double appel — le JSON
est échappé une seule fois dans `injectLensBudgetIntoJbakeProperties`.
Mais la fonction ne protège pas contre un usage futur incorrect.

*Fix* : Documenter "cette fonction n'est PAS idempotente" dans le KDoc
+ ajouter un test qui démontre la corruption en double appel.

*Time* : 5 min.

=== Points de vigilance architecturale

==== Couplage BakeryPlugin → FileSystemManager.yamlMapper

`BakeryPlugin.kt:93` utilise `yamlMapper.readValue<SiteConfiguration>(configFile)`
directement dans le flux `apply()`. Le YAML est parsé pour décider du
branchement scaffold/fullPipeline. Si le YAML est invalide, `readValue`
jette et le plugin ne s'enregistre pas.

*Recommandation* : Wrapper le parse dans un `Either.catch` comme le fait déjà
`FileSystemManager.read()` — mais `read()` retourne un objet par défaut, ce
qui n'est pas adapté ici (on veut vraiment savoir si le site est valide).

==== Pas de parallélisme entre tâches Gradle

Toutes les tâches du pipeline full sont séquentielles. Certaines sont
indépendantes et pourraient être parallélisées :
- `collectSiteContext` || `generateTheme` (pas de dépendance mutuelle)
- `pagefind` || `validateFirebaseConfig` (pas de dépendance mutuelle)

*Recommandation* : Ajouter `mustRunAfter` / `shouldRunAfter` pour documenter
les relations, et laisser Gradle décider du parallélisme automatique.

==== `SiteManager.configureConfigPath` — side-effect non traçable

La méthode modifie `bakeryExtension.configPath` en side-effect sans retourner
de valeur. Si le `gradle.properties` est mal formé, l'utilisateur n'a aucun
feedback.

*Recommandation* : Retourner un `Either<String, Unit>` ou au minimum logger
le résultat (succès/échec).

== Risques Identifiés

=== Risque Technique — Templates Complexes

Les templates réels ont des structures **non couvertes par les tests synthétiques** :

- Expressions Thymeleaf imbriquées (`th:with`, `${content.rootpath}`, `th:if`)
- Mix français/anglais (titres en français, termes techniques en anglais)
- Textes dans des attributs complexes (`th:attr="aria-label=#{...}"` combiné avec d'autres attributs)
- Sections conditionnelles (`th:if`, `th:unless`)
- Boucles (`th:each`) avec texte hardcodé à l'intérieur
- Fragments (`th:fragment`, `th:replace`, `th:insert`)

=== Risque Fonctionnel — Faux Positifs/Négatifs

L'extraction automatique peut :

1. **Manquer du texte** (faux négatifs) :
   - Texte dans des `<script>` ou `<style>` (déjà exclu, mais à vérifier)
   - Texte dans des attributs non-standard (ex: `data-tooltip`, `data-label`)
   - Texte concaténé dynamiquement

2. **Extraire du texte à tort** (faux positifs) :
   - Noms propres (Magic Stick, cccp.education, Cheroliv) → ne doivent PAS être traduits
   - Termes techniques (USB, CSS, Bootstrap, GitHub) → ne doivent PAS être traduits
   - URLs, chemins de fichiers, identifiants

3. **Générer des clés inappropriées** :
   - Clés trop longues ou avec caractères spéciaux
   - Doublons de clés entre templates
   - Clés non-sémantiques (`header.1`, `footer.23`)

=== Risque Opérationnel — Dry-Run Non Validé

Le mode `dryRun=true` (par défaut) n'a **JAMAIS été testé sur un vrai site** :

- Le rapport d'extraction est-il lisible ?
- Le nombre de clés extraites est-il cohérent ?
- L'utilisateur peut-il valider l'extraction avant de lancer la migration réelle ?

== Architecture de la Migration

=== Phases de la Tâche `migrateToI18n`

[source,kotlin]
----
I18nMigrationService.migrate(
    siteDir = File("office/sites/magic-stick/jbake"),
    languages = listOf("fr", "en", "es", "de", "it", "pt", "nl", "pl", "ar", "zh"),
    defaultLanguage = "fr",
    dryRun = true  // PAR DÉFAUT — SÉCURITÉ
)

// Phase 1: scanTemplates() → List<File>
//   └─ Walk récursive du répertoire templates/
//   └─ Filtre *.thyme
//
// Phase 2: extractHardcodedText(templateFile) → Map<key, value>
//   └─ 6 extracteurs :
//      1. Text content: <p>Texte</p>
//      2. placeholder: placeholder="Votre nom"
//      3. aria-label: aria-label="Fermer"
//      4. meta content: <meta content="Description">
//      5. alt: alt="Logo"
//      6. title: title="Accueil"
//   └─ Skips : th:text="#{...}, th:placeholder, th:attr, th:value, <script>, <style>
//
// Phase 3: generateMessageFiles(allExtractions, languages, templatesDir)
//   └─ messages_fr.properties : valeurs originales françaises
//   └─ messages_*.properties : valeurs vides (à traduire)
//
// Phase 4: replaceHardcodedWithMessageKeys(templateFile, extractions)
//   └─ Remplace texte par th:text="#{key}"
//   └─ Remplace placeholder par th:placeholder="#{key}"
//   └─ etc. (6 types)
//
// Phase 5: writeMessageFiles(messageFiles, allExtractions)
//   └─ Écrit les fichiers .properties dans templates/
//
// Phase 6: injectSiteLanguage(siteDir, defaultLanguage)
//   └─ Ajoute site.language=fr dans jbake.properties
----

== Décision Architecturale — US-I18N-MIG-4

=== Traduction LLM — Hors Périmètre Bakery

**DÉCISION (Session 127, 2026-06-16)** : La traduction automatique des 9 langues non-françaises
via LLM (koog+langchain4j) est **hors périmètre bakery-gradle**.

**Raisons** :
1. Capacité transverse : needed by document-gradle, slider-gradle, etc.
2. Dépend de codebase-gradle (N1) : RAG pgvector, LLM pool, koog StateGraph
3. Bakery = expert JBake, pas expert traduction IA

**Solution** :
- `messages_fr.properties` : valeurs françaises (extraction automatique)
- `messages_*.properties` : valeurs **vides** (à remplir manuellement ou via service externe)
- Service de traduction transverse à implémenter dans **codebase-gradle N1** ou **N0 dedicated**

**Impact** :
- US-I18N-MIG-4 (3 pts) = ⏸️ EN ATTENTE
- Bakery consommera ce service via DSL ou CLI
- Migration i18n **bloquent sans traduction** : un site avec 9 fichiers `.properties` vides est inutilisable

== Risques Résiduels

=== Risque P0 — Traductions Manuelles Non Faites

Si US-I18N-MIG-4 (service de traduction transverse) n'est pas implémenté,
les traductions des 9 langues non-françaises doivent être faites **manuellement**.

**Impact** :
- messages_en.properties : ~150-200 clés à traduire (FR→EN)
- messages_es.properties : ~150-200 clés (FR→ES)
- ... × 8 langues = **1200-1600 traductions manuelles**

**Coût estimé** :
- FR→EN : 2-3 heures (révision humaine requise)
- FR→8 autres langues : 16-24 heures (ou service LLM transverse)

=== Risque P1 — Faux Positifs dans l'Extraction

Si l'extraction automatique inclut des noms propres ou termes techniques,
il faudra :
1. Corriger les extracteurs (US-VAL-3)
2. Re-lancer la migration
3. Re-vérifier toutes les clés

**Atténuation** : Dry-run + audit manuel avant migration réelle.

=== Risque P2 — Régression Visuelle

Si Thymeleaf ne résout pas correctement les clés `#{...}`, le rendu peut :
- Afficher les clés brutes (`#{header.title}` au lieu de "Magic Stick")
- Afficher des valeurs vides
- Crash au rendu

**Atténuation** : Test de rendu FR/EN avant déploiement (US-MIG-3, US-GEN-1, US-GEN-2).

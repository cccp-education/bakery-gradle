== Recommandation Stratégique : Finir bakery d'abord ?

**Verdict : OUI — finis bakery. 1 jour. Puis passe au suivant.**

=== Pourquoi finir bakery maintenant

1. **Contexte chaud** — 66 sessions d'accumulation. Switcher = perte 20-30 min de recontextualisation.
2. **EPIC 6 est un fantôme** — le code PublishProfileTask existe. C'est 1h de vérification, pas 13 pts.
3. **BKY-IA-2 a son pattern** — slider-gradle fait déjà du thème paramétrique. Copier-coller adapté.
4. **Aucun blocage externe** — training-gradle, runner-gradle, etc. ne dépendent pas de bakery.
5. **Jalon psychologique** — finir un borough proprement = momentum pour le suivant.

=== Pourquoi NE PAS finir bakery maintenant (contre-arguments)

- Diminishing returns : les EPICs restants sont du polish (coverage 73%→75%, thème IA paramétrique)
- Si un autre borough a une **deadline externe** (client, démo), priorité différente
- L'ennui : si les EPICs IA ne te motivent pas, le rendement sera faible

=== Recommandation : Scénario C (split)

== Code Smells & Mauvais Patterns

=== 🔴 CRITIQUE — À corriger avant finalisation

// tag::CS1[]
==== CS-1 : God Class SiteManager — 874 lignes (+99% depuis audit 022)

*Problème* : `SiteManager.kt` (873 lignes, voir `AGENTS.adoc` section "Degats Collateraux 3")
concentre TOUTE la logique du plugin :
- 31+ méthodes (register*, inject*, configure*, setup*, etc.)
- Task registration, config injection, git push, pagefind, serve, generateArticle,
  generateSiteFromIntention, collectSiteConfig, collectSiteContext, collectAugmentedContext
- 8 registres de tâches mélangés + logique métier inline dans les lambdas `doLast`

*Pattern violé* : Single Responsibility Principle

*Risque* : lisibilité dégradée, merge conflicts probables, testabilité réduite.

*Recommandation* : Extraire 4 TaskRegistrars spécialisés :
- `SiteTaskRegistrar.kt` → generateSite, serve, collectSiteConfig
- `DeployTaskRegistrar.kt` → deploySite, deployMaquette, deployProfile
- `ContentTaskRegistrar.kt` → generateArticle, generateSiteFromIntention
- `LensTaskRegistrar.kt` → collectSiteContext, collectAugmentedContext

*Contre-argument* : Le pattern "objet SiteManager unique" est utilisé par plantuml-gradle (même approche).
Ce n'est pas un défaut absolu tant que le nombre de méthodes reste < 40. À 874 lignes, c'est
un point de vigilance, pas un blocage de prod.
// end::CS1[]

// tag::CS2[]
==== CS-2 : Logique métier inline dans les lambdas Gradle `doLast`

*Problème* : `registerCollectAugmentedContextTask()` (lignes 747-838, 91 lignes)
contient l'intégralité du pipeline LENS dans un bloc `doLast { }` :
- Chargement composite-context
- Extraction sous-graphe BFS
- Scoring hybride (AugmentedArticlesService)
- Application budget (LensBudget)
- Sérialisation JSON en dur

*Pattern violé* : Une tâche Gradle devrait déléguer, pas orchestrer 5 services.

*Recommandation* : Extraire dans un `LensPipelineService` :
```kotlin
class LensPipelineService {
    fun execute(config: AugmentedContextDsl): AugmentedOutput
}
```
Le `doLast` deviendrait 3 lignes : `val output = pipeline.execute(config); output.writeTo(dir)`.
// end::CS2[]

// tag::CS3[]
==== CS-3 : Parse JSON maison fragile (ScaffoldGenerator)

*Problème* : `ScaffoldGenerator` utilise un parse JSON maison (`extractJson`, `extractJsonValue`,
`extractJsonArray`) plutôt que Jackson (déjà dans le classpath).

*Risque* : Fragile aux variations de format LLM (whitespace, unicode, nested objects).
Non testé exhaustivement sur les edge cases.

*Recommandation* : Remplacer par Jackson :
```kotlin
val mapper = jacksonObjectMapper()
val json = extractJsonBlock(text)
val output = mapper.readValue<ScaffoldOutput>(json)
```
// end::CS3[]

=== 🟠 MODÉRÉ — Refactoring souhaitable à moyen terme

// tag::CS4[]
==== CS-4 : Injection config monolithique (70 lignes pour 7 domaines)

*Problème* : `injectResolvedConfigIntoJbakeProperties()` (lignes 234-305 dans SiteManager)
gère 7 domaines (Firebase, GoogleForms, FirebaseAuth, Comments, Analytics, Newsletter,
Theme, Layout) dans un seul `when` block + 7 appels `updateProperty`.

*Recommandation* : Pattern Strategy — `Map<String, ConfigInjector>` :
```kotlin
fun interface ConfigInjector {
    fun inject(lines: MutableList<String>, resolver: (String, String) -> String)
}
```
// end::CS4[]

// tag::CS5[]
==== CS-5 : Copies conformes ConfigResolver (7+ méthodes quasi-identiques)

*Problème* : `ConfigResolver` a 9 méthodes resol* (GoogleForms, Analytics, FirebaseAuth,
Comments, Newsletter, Theme, Layout, Firebase, Enum) — quasi-identiques, ~200 lignes de copier-coller.

*Recommandation* : Approche générique `inline fun <reified T> resolve()` avec sealed class
ou registry pattern. Réduction estimée : 200 → 50 lignes.
// end::CS5[]

// tag::CS6[]
==== CS-6 : BakeryExtension — 11 DSL blocks manuels

*Problème* : Chaque sous-config suit le même pattern :
```kotlin
val xxx: XxxDsl = XxxDsl()
fun xxx(action: Action<XxxDsl>) { action.execute(xxx) }
```
11 blocks × 2 lignes/block = 22 lignes de boilerplate + 11 data classes × ~30 lignes = ~350 lignes.

*Recommandation* : Annotation processor `@DslBlock` ou pattern registration via Map.
Ou accepter — c'est du Kotlin idiomatique, pas une dette technique critique.
// end::CS6[]

// tag::CS7[]
==== CS-7 : BKY-IA-0 (LLM service) non dépilé vers codebase — divergence doc↔code

*Problème* : La roadmap `BAKERY_ROADMAP_JBake.adoc` dit :
> IA centralisée : codebase (N1) est le SEUL provider IA. BKY-IA-0 (LLM service) à dépiler.
Mais `OllamaLlmService` + `LlmService` + `IaConfig` sont toujours dans bakery.

*Risque* : Divergence documentation ↔ réalité. Deux providers IA coexistent sans coordination.

*Recommandation* : Soit dépiler réellement (retirer LLM service, appeler codebase),
soit mettre à jour la roadmap. Décision : **garder dans bakery** — le contrat DAG N2 autorise
un LLM interne, pas de dépendance à N1. Simple mise à jour de la roadmap.
// end::CS7[]

=== 🟡 LÉGER — Vigilance, pas bloquant

// tag::CS8[]
==== CS-8 : Parse XML RSS/DOM fragile

*Problème* : `SiteContextCollector.parseFeed()` utilise `DocumentBuilderFactory` + DOM traversal
manuel. Si JBake change le format de feed.xml, ça casse silencieusement.

*Recommandation* : Rome library pour RSS ou JAXB pour sitemap. Priorité basse — JBake 2.7.0
est stable depuis 2020.
// end::CS8[]

// tag::CS9[]
==== CS-9 : `var` dans les data classes du package `lens`

*Problème* : `LensConfig` et `LensBudget` utilisent `var` pour toutes leurs propriétés.
Perte des garanties d'immutabilité et de thread-safety.

*Recommandation* : Remplacer `var` par `val` + `copy()`. Priorité basse — ces classes sont
utilisées uniquement en configuration séquentielle.
// end::CS9[]

// tag::CS10[]
==== CS-10 : Tasks TODO qui `throw NotImplementedError`

*Problème* :
```kotlin
tasks.register("createPagesRepository") {
    doLast { throw NotImplementedError("planned but not yet implemented") }
}
```
Ces tâches sont enregistrées mais cassent à l'exécution. L'utilisateur les découvre via
`./gradlew tasks` et tombe sur une erreur.

*Recommandation* : Soit implémenter, soit `@Deprecated` + message. Soit retirer
(le GitHub API token est une feature future).
// end::CS10[]

// tag::CS11[]
==== CS-11 : Logger incohérent — `println()` + `logger.info()` mélangés

*Problème* : Certaines méthodes utilisent les deux :
```kotlin
"create config file.".apply(::println).let(logger::info)
```

*Risque* : Les `println()` sont visibles en sortie même en `--quiet`.

*Recommandation* : Uniformiser vers `logger.lifecycle`. Priorité basse — c'est cosmétique.
// end::CS11[]

// tag::CS12[]
==== CS-12 : Inconsistance historique : publishProfile vs deployProfile

*Problème* : La tâche est nommée `deployProfile` dans le code mais `publishProfil` (faute d'orthographe
"Profil" au lieu de "Profile") dans BACKLOG et INDEX. Le fichier `PublishProfileTask.kt` garde
le nom historique `publishProfile()`.

*Recommandation* : Renommer la méthode `PublishProfileTask.publishProfile()` → `deployProfile()`
pour alignement taxonomique K-1. Priorité basse (non bloquant).
// end::CS12[]

DDD/TDD/BDD est exemplaire, les 4 couches de test couvrent tout le spectre,

**Le vrai chantier restant est minuscule : ~17 pts, soit 1 jour de travail.**

CS-5 ConfigResolver verbeux) est documentée et non bloquante pour la production.
Le produit peut être livré et utilisé tel quel.

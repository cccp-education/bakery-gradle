== Points Forts

=== 1. Méthodologie DDD/TDD/BDD rigoureuse

Chaque EPIC suit les Baby Steps :
. **DDD** — data classes domaine (`XxxConfig`, `XxxDsl`)
. **TDD** — tests unitaires (`XxxConfigTest`, `XxxTemplateTest`, `XxxInjectionTest`)
. **BDD** — Cucumber scénarios (`.feature` + `XxxSteps`)

Ce pattern est reproductible et systématique. On le retrouve sur :
- Google Forms (JB-3) ✅
- Firebase Auth + Comments (JB-4) ✅
- Analytics + Newsletter (JB-5) ✅
- Theme (JB-6) ✅
- Layouts (JB-7) ✅
- Article IA (JB-8) ✅

=== 2. Architecture Clean — ArticleGenerator

[source,kotlin]
----
// ArticleGenerator.kt — Pure logique métier, zéro dépendance Gradle
class ArticleGenerator {
    suspend fun generate(intention: ArticleIntention, llm: LlmService): ArticleOutput
}
----

- `ArticleIntention` (DDD) encapsule topic/ton/audience/keywords/lang
- `LlmService` est une `fun interface` — testable, mockable
- `ArticleOutput` est une `data class` pure
- Le parseur `parseResponse()` est résilient (fallback article en cas d'échec)

=== 3. ConfigResolver — Architecture de surcharge 4 couches

[source]
----
CLI (-P) > gradle.properties > DSL (BakeryExtension) > site.yml (YAML) > defaults
----

- 50 tests unitaires ✅
- Types supportés : `String`, `Int`, `Boolean`, `Enum`
- Méthodes composées : `resolveGoogleFormsConfig`, `resolveAnalyticsConfig`, etc.
- Inspiré de plantuml-gradle ConfigMerger, simplifié avec Map-based approach

=== 4. Qualité des tests

| E2E Playwright | 13 tests navigateur (Chromium + Cucumber)

|===

=== 🔴 CRITIQUE 1 — SiteManager.kt : god class (819 lignes)

.Diagnostic
[source]
----
Audit 022 (29 mai) : SiteManager.kt = 439 lignes (31% du src)
Maintenant (2 juin) : SiteManager.kt = 819 lignes — PRESQUE DOUBLÉ

9 méthodes `inject*IntoJbakeProperties` quasi identiques (copier-coller) :

[source,kotlin]
----
private fun Project.injectXxxConfigIntoJbakeProperties(targetDir: File, site: SiteConfiguration) {
    val jbakeProps = targetDir.resolve(site.bake.srcPath).resolve("jbake.properties")
    if (!jbakeProps.exists()) { logger.warn(...); return }
    val config = site.xxx ?: return
    val lines = jbakeProps.readText(UTF_8).lines().toMutableList()
    fun updateProperty(key: String, value: String) { ... }
    updateProperty(...)  // 2-6 clés spécifiques
    jbakeProps.writeText(lines.joinToString("\n"), UTF_8)
}
----

*Impact* : maintenabilité dégradée, risque de régression si une méthode oublie une clé.

.Recommandation
. Extraire une méthode générique :
+
[source,kotlin]

private fun Project.injectConfigIntoJbakeProperties(
    targetDir: File, site: SiteConfiguration,
    properties: Map<String, String>
)

. Splitter `SiteManager` en `TaskRegistrar` délégués par domaine :
  `BakeTaskRegistrar`, `DeployTaskRegistrar`, `ProfileTaskRegistrar`

=== 🟠 GRAVE 4 — Dépendance graphify-gradle non utilisée

`build.gradle.kts` ligne 34 : `implementation(libs.graphify.plugin)`.

Bakery a construit **son propre KG** (`RelatedArticlesService`, `RelatedArticlesGraph`,
`ArticleNode`) sans utiliser graphify. La dépendance alourdit le classpath sans servir.

*Impact* : dette technique silencieuse. La classe `GraphConsumptionSteps.kt` existe
mais le test n'est pas connecté à une vraie consommation de graphify.

.Recommandation
. Vérifier si `graphify` est utilisé quelque part (grep sur tout le code source)
. Si non : retirer la dépendance et le `GraphConsumptionSteps` correspondant
. Si oui : documenter l'usage dans `AGENT.adoc`

[%hardbreaks]
Bakery-gradle est un plugin mature et complet dans son périmètre d'expert métier JBake.

bien testé, et bien gouverné.

d'assemblage de la formation. Il est le moteur de site statique que les autres boroughs
utilisent pour déployer leurs artefacts.

**Règle d'or** : *CONV-1.4 (migration ConfigResolver)* avant toute nouvelle feature.
La dette d'architecture est le seul frein à la vélocité.

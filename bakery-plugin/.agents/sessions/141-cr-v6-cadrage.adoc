== Audit — 3 duplications structurelles confirmées

=== Duplication 1 — Pattern `resolveIntention()` dupliqué ×4

Fichiers :
- `GenerateArticleTask.kt:178` — `resolveIntention(): ArticleIntention`
- `GenerateThemeTask.kt:205` — `resolveIntention(): ThemeIntention`
- `GenerateSiteFromIntentionTask.kt:160` — `resolveIntention(): ScaffoldIntention`
- `MigrateToI18nTask.kt:96` — `resolveIntention(): I18nMigrationIntention`

23 occurrences du pattern `orNull?.takeIf{isNotBlank} ?: dslIntention?.field ?: default`.
Squelette identique : résolution CLI > DSL > defaults.
`ContentTaskRegistrar.resolveIntention` (ligne 211) existe mais ne couvre que
l'injection DSL dans la tâche, pas la résolution CLI dans la tâche elle-même.

BACKLOG V5 US-CR-V5-5 mentionnait « Pattern résolution CLI>DSL>defaults
dupliqué 3× → extraire `ResolveIntention<T>` helper » — marqué ✅ S124.
**Le helper n'existe pas dans le code.** Le ✅ est faux.

=== Duplication 2 — DomainConfigInjectors 9 injecteurs répétitifs (113 lignes)

`DomainConfigInjectors.kt` : 9 objects avec même forme :
```kotlin
override fun inject(lines, resolver) {
    val x = resolver("key", "")
    if (x.isNotBlank()) { updateProperty(lines, "key", x); ... }
}
```
3 variantes :
- gate sur champ trigger (Firebase, GoogleForms, FirebaseAuth, Analytics)
- gate sur boolean (Comments, Newsletter)
- injection systématique (Theme)

Data-driven possible via `InjectorSpec(domain, trigger, mode, properties)`.

Non identifié dans les audits V4/V5 précédents. Nouveau finding.

=== Duplication 3 — ConfigResolver.resolveAll() 10 blocs `resolveDomain`

`ConfigResolver.kt:332-401` : 10 blocs identiques :
```kotlin
val resolvedX = resolveDomain("x", { resolveXConfig(props, ...) }, XConfig())
```
BACKLOG V5 US-CR-V5-7 disait « `resolveAll()` 9 blocs `resolveDomain` répétitifs
→ data-driven (liste de DomainResolver) » — marqué ✅ S123.
**Le code actuel ligne 332-401 n'a pas bougé.** Le ✅ est faux.

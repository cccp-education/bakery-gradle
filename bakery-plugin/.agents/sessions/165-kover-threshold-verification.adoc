=== 4. Cadrage EPIC BKY-TREE — Arbre de Site Composite

Suite à la découverte que `SiteConfiguration` est plate (pas d'arborescence
éditoriale), l'utilisateur a proposé le concept unifiant manquant : `SiteTree`
(Composite Site→Section→Article→PivotModel AST).

* *Motivation* : `SiteConfiguration` est plate (infra only). `ScaffoldOutput.
  templates: List<String>` est plate. La migration i18n walk `templates/` à
  plat. Aucun concept ne modélise l'arborescence éditoriale. Conséquence : la
  Loi de l'Économie d'Encre ne peut pas calculer un delta ciblé par sous-arbre.
* *Vision* : agregat `SiteTree` à 4 niveaux — `Site` (root) → `Section` →
  `Article` (feuille) → `PivotModel` (AST déjà construit S157-163). Chaque
  niveau manipulable indépendamment (theme, i18n, accessibilité, Économie
  d'Encre par sous-arbre).
* *7 US cadrées* : TREE-1 (SiteNode sealed, 5 pts) → TREE-2 (SiteTree aggregate,
  5) → TREE-3 (Content wrapper PivotModel, 3) → TREE-4 (YAML serialization,
  5) → TREE-5 (Theme inheritance, 5) → TREE-6 (i18n par sous-arbre, 8) →
  TREE-7 (Scaffold IA arbre, 5). Total ~36 pts.
* *Projection* : S166 TREE-1+2 (socle domaine pur), S167 TREE-3+4 (AST+YAML),
  S168 TREE-5+6 (theme+i18n), S169 TREE-7 (scaffold, clôture EPIC).
* *Pré-requis* : ne pas réinventer PivotModel (fait, testé, golden-master) ;
  backward compat (`templates: List<String>` reste accepté via adapter) ;
  baby steps DDD/TDD/BDD à chaque US.

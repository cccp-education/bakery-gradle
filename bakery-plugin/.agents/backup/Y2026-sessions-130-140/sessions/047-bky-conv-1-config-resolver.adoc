== Constats et Décisions

1. **Audit cross-borough** : plantuml-gradle a un ConfigMerger explicite à 5 couches (CLI > ENV > gradle.properties > YAML > defaults). codebase-gradle utilise `providers.gradleProperty().orElse(extension)`. slider-gradle est YAML-centric avec override CLI ponctuel.
2. **Décision architecture** : Bakery adopte 4 couches (CLI+gradle.properties > DSL > YAML > defaults) car ENV vars ne sont pas pertinentes pour un plugin Gradle build-time.
3. **Pattern innovant** : `dslValue != default` comme heuristique pour détecter qu'un DSL a été explicitement configuré. Permet à un DSL vide de "tomber" vers YAML, et à un DSL configuré de surcharger YAML.
4. **9/10 configs n'ont aucun override** : Seul `articleIntention` a un vrai mécanisme CLI > DSL > defaults via `GenerateArticleTask.resolveIntention()`.

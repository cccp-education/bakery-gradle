= Session 054 — Audit Intégration Réalité Augmentée dans Bakery (Pattern LENTILLE)

_Date_ : 2026-06-02
_Type_ : Audit + Conception architecture
_Score_ : N/A (audit, pas de code modifié)

== Contexte

Audit de l'intégration du pipeline de réalité augmentée (graphify + codebase) dans bakery-gradle. La session part d'une question fondamentale : _ce n'est plus graphify qu'il faut intégrer, mais toute la réalité augmentée portée par codebase-gradle (N1)._

== Constats — 3 Problèmes Identifiés

=== 🔴 Problème 1 : Bakery a réinventé la roue du Knowledge Graph

Bakery a construit un système KG interne (`RelatedArticlesService`, `RelatedArticlesGraph`, `ArticleNode`, `RelatedArticlesResolver`) avec un scoring propriétaire :

- Tags : `1.0/tag` (recouvrement littéral)
- Titres : `0.2/mot` (mots partagés)
- Co-occurrences : `0.3/count` borné
- Entités : `0.3/entity` (TF-IDF sur descriptions)

Ce scoring est **strictement inférieur** à ce que le pipeline codebase+graphify produit déjà :

| Capacité | Bakery interne | Pipeline augmenté codebase+graphify |
|----------|---------------|--------------------------------------|
| Tags | `1.0/tag` manuel | RAG pgvector Similarité sémantique (`0.89+`) |
| Titres | `0.2/mot` mot-à-mot | Embeddings 384-dim, similarité cosinus |
| Co-occurrences | `0.3/count` borné | Graphify `203k nodes, 188k edges` couplage structurel |
| Entités | TF-IDF sur descriptions | CodexVectorStore corpus documentaire complet |
| Cross-borough | Aucun | Composite context (codex + training + codebase) |
| Budget tokens | Aucun contrôle | `ChannelBudget` 8000 tokens, 40/30/20/10 |

=== 🔴 Problème 2 : Dépendance graphify déclarée mais morte

`build.gradle.kts` L34 : `implementation(libs.graphify.plugin)` — **zéro `import` dans tout le code source Kotlin**. Le code review GRAVE 4 l'a signalé, la session 050 l'a restauré pour « BKG nécessaire », mais **rien n'est câblé**.

Pendant ce temps, codebase-gradle (N1) consomme déjà graphify via le canal `ContextChannel.Graphify` (`office/graph.json`, 20% du budget tokens).

=== 🟡 Problème 3 : Bakery n'est pas connecté au pipeline N3

L'orchestrateur `pilot-plugin` (N4) appelle `assembleCompositeContext` qui agrège codex + training + codebase, mais **bakery n'en fait rien**. Le `composite-context.json` et `metadata.json` sont produits mais jamais injectés dans le site.

== Décision Architecturale — Pattern LENTILLE

Les deux options proposées (ségrégation de nœuds vs recréation de contexte) ne sont pas opposées — elles sont **complémentaires**. Le pattern LENTILLE combine les deux en couches :

. **Ségrégation** : filtrer les 203k nœuds de `graph.json` en un sous-graphe ciblé (~200 nœuds)
. **Enrichissement** : ajouter les edges RAG (similarité sémantique) + règles métier
. **Budget** : tronquer au nombre max d'articles/page, similarité min, tokens max

=== 3 Couches, 3 Responsabilités

| Couche | Qui | Quoi | Quand
| Ségrégation | `SubgraphExtractor` (bakery) | Filtre graph.json par communauté + type d'edge + depth de voisinage | À `collectRelatedArticles`
| Enrichissement | `AugmentedArticlesService` (bakery) | Ajoute edges RAG (similarité sémantique) + règles métier | À `collectRelatedArticles`
| Budget | `LensBudget` (N0 contract) | Tronque à N articles/page, similarité min | À l'injection JBake

=== DSL Proposée

```kotlin
bakery {
    augmentedContext {
        enabled = true

        lens {
            // SÉGRÉGATION — quelles parties du graphe ?
            scope = SUBGRAPH                    // SUBGRAPH | FULL | SEMANTIC_ONLY
            communities = ["bakery-gradle",
                           "codebase-gradle",
                           "slider-gradle"]
            nodeTypes = ["file"]                 // ne garder que les fichiers
            edgeTypes = ["reference", "agent_reference"]  // pas "contains"
            maxDepth = 2                         // 2 sauts depuis la page courante
            fileExtensions = ["adoc", "md", "html"]  // filtrer les sources éditoriaux

            // ENRICHISSEMENT — quelles sources ajouter ?
            rag {
                enabled = true
                similarityThreshold = 0.7       // similarité minimale
                topK = 20                         // candidats RAG avant filtrage
            }

            // RÈGLES MÉTIER — filtres éditoriaux
            rules {
                excludeDrafts = true               // exclure les articles statut=brouillon
                excludeTags = ["wip", "draft"]     // exclure par tags
                prioritizeCrossReferences = true    // les xref AsciiDoc ont priorité
                crossRefBonus = 0.2                // bonus de score pour les xref
                communityAffinity = 0.3            // bonus pour les articles du même borough
            }

            // BUDGET — combien montrer ?
            budget {
                maxArticlesPerPage = 4             // articles connexes par page
                maxTokensPerArticle = 200           // résumé max par article
            }
        }
    }
}
```

=== Scoring Hybride — Formule

Pour chaque article candidat, le score final combine 4 signaux :

----
score = α × ragSimilarité      // 0.40 — similarité sémantique (pgvector)
      + β × graphProximité     // 0.30 — voisinage structurel (graphify depth)
      + γ × tagRecouvrement   // 0.15 — tags éditoriaux partagés (JBake metadata)
      + δ × crossRefCount     // 0.15 — références croisées AsciiDoc (xref)

Où :
  α = 0.40, β = 0.30, γ = 0.15, δ = 0.15

  + bonus 0.2 si même communauté (communityAffinity)
  + bonus 0.2 si cross-reference AsciiDoc vérifiée (prioritizeCrossReferences)
----

=== Pourquoi Pas « Juste Filtrer le Graphe »

Le **bruit sémantique** ne se voit pas dans la structure. Exemple :

article: "Installer pgvector sur Ubuntu 24.04"

Ségrégation seule (2-hop dans graph.json) :
  → "Ubuntu 24.04 release notes" (même communauté Linux)
  → "Kitty Terminal config" (même fichier Makefile)
  → "Docker compose pour integration tests" (même repo)

Ségrégation + RAG + règles (lentille hybride) :
  → "Embeddings 384-dim avec ONNX" (similarité 0.91)
  → "CodexVectorStore recherche sémantique" (similarité 0.88)
  → "ChunkTokenizer découpage overlap" (similarité 0.82)
  → "PertinenceBenchmark qualité RAG" (similarité 0.79)

Les 3 résultats de la ségrégation seule sont structurellement proches mais sémantiquement sans intérêt. Les 4 résultats du pipeline hybride sont exactement ce que le lecteur veut voir ensuite.

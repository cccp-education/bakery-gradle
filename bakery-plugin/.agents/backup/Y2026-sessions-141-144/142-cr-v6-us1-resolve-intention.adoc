== Audit du pattern (DDD caractérisation)

Les 4 `resolveIntention()` lus en parallèle. Pattern commun :

    cliField.orNull?.takeIf{isNotBlank} ?: dslIntention?.dslField ?: default

5 variantes identifiées :
- String simple (description, lang, projectName, defaultLang, ton, audience)
- List<String> (keywords, langs — split/filter/takeIf isNotEmpty)
- Nullable (ThemeOverrides 7 champs — pas de default)
- Throw si absent (topic, siteDir, scaffoldDescription)
- Boolean (dryRun — toBooleanStrictOrNull)

Safety net vérifié : 4 `*TaskTest.kt` existants couvrent déjà `resolveIntention()`.
Baseline GREEN établie avec permission utilisateur (toutes 4 Task PASS).

== Modélisation FP / Arrow-kt

Arrow 2.2.2 — `Option` dépréciée en Arrow 2.x (recommandation : nullable Kotlin +
`Either`). Code bakery existant cohérent (uniquement `Either`/`left`/`right`).

Choix modélisation :
- `fromCli`, `fromCliNullable`, `fromCliList`, `fromCliBoolean` : nullable Kotlin
  (cascade `?:` idiomatique, pas de monade inutile).
- `fromCliRequired` : `Either<ResolveIntentionError, String>` (erreur explicite
  via sealed class, composition FP `.fold(ifLeft={throw it.toException()},
  ifRight={it})`).

Sealed class `ResolveIntentionError` (introduite sur demande utilisateur —
pertinent pour normaliser les messages d'erreur, DDD erreur explicite au lieu
d'`IllegalArgumentException` générique) :
- `MissingRequiredField(cliFlag, dslPath)` + `toException()` remontée au niveau
  parent sealed (abstract) pour accès via `fold` sans cast.
